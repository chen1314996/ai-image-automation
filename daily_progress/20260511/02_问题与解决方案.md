# 问题与解决方案 - 2026-05-11

## 问题 1：创意拓展停止后无法明确选择继续或新任务

### 现象
用户停止创意拓展工作流后，希望可以随时继续之前任务，也可以开启新任务，但页面缺少清晰选择。

### 原因
旧流程只关注“开始/停止”，没有把“停止后的恢复状态”作为一等状态呈现给用户。

### 解决方案
- 增加“继续任务”和“新任务”按钮。
- 继续任务依赖后端恢复状态。
- 新任务用于忽略旧恢复状态并重新开始。
- 按钮视觉保持低调，避免干扰主操作。

### 结果
停止后用户可明确选择下一步，不需要重新理解当前状态。

## 问题 2：继续任务按钮默认样式与可点击时机不符合预期

### 现象
用户要求继续任务默认与新任务一样是深灰色，且只有停止工作流后才能点击。

### 原因
初始设计容易把“继续任务”当成主按钮，但用户希望它是停止后的状态操作。

### 解决方案
- 继续任务默认使用深灰色次级按钮。
- 没有停止/中断状态时禁用。
- 停止后才启用。

### 结果
按钮状态更符合“恢复任务”的语义。

## 问题 3：保存图片时浏览器窗口会抢前台

### 现象
用户最小化浏览器后，保存图片阶段窗口又弹出。

### 原因
有头模式下，Playwright 点击缩略图、打开预览图、截图或下载时可能触发浏览器窗口激活。

### 尝试
- 尝试减少保存阶段窗口激活。
- 用户反馈弹出更频繁。

### 最终策略
- 恢复原有行为，避免破坏保存功能。
- 使用无头模式解决“不希望弹窗”的核心诉求。
- 保留有头模式用于登录、观察和手动排查。

## 问题 4：Agent 在项目中生成的表格质量低于 ChatGPT 调用效果

### 现象
用户提供三份表格：
- 原始输入表格
- 项目 Agent 生成表格
- ChatGPT 调用 Agent 生成表格

项目生成效果明显更粗略，详细程度不足。

### 初步判断
项目调用 Agent 时可能存在以下差异：
- 输入上下文传递不足。
- 系统提示词约束不够强。
- 表格结构要求不够明确。
- 目标数量、方向细节、提示词组数未被严格执行。
- 缺少质量自检和二次扩写。

### 优化方向
- 对输入表格进行结构化解析，而不是只传文件摘要。
- 增加 Agent 输出质量评分。
- 强制每个方向包含标题、说明、5 组提示词。
- 对缺失字段进行二次补全。
- 将 ChatGPT 调用结果作为目标质量样例。

## 问题 5：API Key 安全风险

### 现象
用户直接提供 Lumos Winky API Key，希望写入后端，不想前端填写。

### 风险
如果写入前端或日志，会造成泄漏。

### 解决方案
- 写入本地忽略文件。
- 后端读取。
- 前端只显示“已配置/未配置”。
- 日志中做脱敏。

### 结果
满足免前端填写需求，同时降低泄漏风险。
