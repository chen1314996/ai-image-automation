# 问题与解决方案 - 2026-05-09

## 今日遇到的问题

### 问题1：独立豆包/Legil 调用被误判为工作流停止

**现象：**
- 单独调用豆包自动化或 Legil 生成时，工作流本来没有运行
- 模块内部读取 `workflowController.isRunning`
- 因 `isRunning === false` 被误判为“工作流已停止”

**根因：**
自动化模块反向依赖工作流单例，取消状态没有由调用方显式传入。

**解决方案：**
- 将取消状态改为通过参数或 `AbortSignal` 传入
- 独立调用和完整工作流调用分离
- 模块内部不再硬依赖全局 workflow 状态

---

### 问题2：前端显示“已启动”，但后端实际启动失败

**现象：**
- 输入目录无效、空目录或重复启动时，接口仍提前返回成功
- UI 状态被误导，后续可能卡住

**根因：**
`/api/workflow/start` 在完整校验前就响应 `success:true`。

**解决方案：**
- 先完成输入目录、重复运行、工作流启动结果校验
- 只有真正启动成功后才返回成功
- 错误信息直接返回给前端

---

### 问题3：中文提示词被过滤

**现象：**
```
过滤掉非英文提示词: 第 1 组：冰封绝境的预警信号...
❌ 提取失败：只找到 0 组有效提示词，需要5组
```

**根因：**
早期逻辑默认提示词应为英文，对中文提示词进行了过度过滤。

**解决方案：**
- 取消“必须英文”的判断
- 改为识别是否包含有效画面描述
- 允许中文、英文、中英混合内容进入 Legil

---

### 问题4：Legil 多图生成误判完成或卡住

**现象：**
- 图片还在生成中，日志已经提示检测到新图片并开始下载
- 输出 4 张时，只能看到前 3 张，系统无法检测第 4 张
- 有失败占位图时一直等待

**根因：**
- 生成完成判断只看按钮或可见图片数量
- 未区分当前生成行和历史图
- 未识别失败占位
- 横向滚动策略过于频繁

**解决方案：**
- 记录生成前后的图片集合，仅保存本次新图
- 当前行前 3 张完成后才定时横向查看第 4 张
- 失败占位默认计入完成槽位但不保存
- 只滚动当前生成行

---

### 问题5：大图下载偶发 HTTP 403

**现象：**
```
大图直链下载失败: HTTP 403
大图页面内下载失败，改用大图截图保存
```

**根因：**
Legil 输出图在 OSS/CDN 上的访问策略不稳定，有些直链可访问，有些需要页面上下文或只能在弹窗中渲染。

**解决方案：**
- 直链下载失败后等待 3 秒重试，最多 3 次
- 连续失败后尝试页面内下载
- 再失败则使用大图截图兜底

---

## 预防措施

1. **模块边界**
   - 自动化模块不直接读取全局 workflow 状态
   - 工作流控制器只负责编排，不把状态隐式散落到子模块

2. **生成结果识别**
   - 生成前必须记录历史图集合
   - 保存时必须校验输出图地址属于本次新图
   - 避免保存参考图、旧图或失败图

3. **前端日志安全**
   - 用户输入路径、错误消息、提示词预览都不得直接拼接 `innerHTML`
   - 优先使用 `textContent`

4. **下载链路**
   - 直链下载不是唯一可靠方案
   - 必须保留页面上下文下载和截图兜底
